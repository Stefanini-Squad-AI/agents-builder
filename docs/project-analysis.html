🤖 Agents Workshop

Meta-automation tool for AI coding agent contracts

MVP Complete (Phase P2)

💡 What It Is

A meta-automation tool that generates the "contract" (skill library + phased Jira backlog) that AI coding agents can execute.

Important Distinction

It's NOT a code generator — it generates the instructions other AI agents use.

👤 Target User

Architects and Engineering Leads bootstrapping new PoCs who want consistent skill+card scaffolding across projects without copy-pasting from previous repos.

✅ Business Problems Solved

  • Eliminates repetitive work — No more manual creation of agent instruction files
  • Standardization — Consistent structure for AI-driven development projects
  • Auditability — Full trail of all LLM interactions (costs, tokens, timing)
  • Reproducibility — PoC setup becomes shareable and consistent

🔄 Discovery → Contract Pipeline

The complete 7-stage process from project creation to exportable agent contract:

# Stage Purpose Implementation
1 Discovery Gather project context 4 channels: Objective, Q&A wizard, Tech panorama, Artifact uploads
2 ProposeSkillSet AI suggests skills LLM prompt with few-shot examples → 5-10 skills
3 DraftSkillBody Generate skill content Per-skill LLM call with kind-specific biases
4 ProposeBacklog Create implementation plan LLM generates 3-7 phases with cards + dependencies
5 DraftCard Generate card bodies Per-card LLM call with 10-section format
6 Validate Ensure graph integrity DAG cycles, reference resolution, required fields
7 Export Generate .agents/ folder Skills + Jira cards as markdown with Mermaid DAG

📋 Business Rules

Skill Rules

  • Four kinds: context, authoring, analyzer, procedure
  • kind='analyzer' with 0 resources → WARNING (not blocking)
  • Skill slug must be unique per project

Card Rules

  • Every card MUST have ≥1 skill link → ERROR if not
  • Card code must be unique per project (format: {PREFIX}-{phase}{card_num})
  • human_gate=true requires non-empty checklist
  • Last card of each phase SHOULD have human_gate=true → WARNING if not

Dependency Rules

  • All depends_on references MUST resolve → ERROR
  • No DAG cycles allowed → ERROR (Kahn's algorithm)
  • Forward-phase dependencies forbidden → ERROR

📚 Artifact Enrichment: Analysis as Enhancement

After raw extraction, AI generates an enriched analysis document that adds business context, user journeys, and decision points — without replacing the source.

Key Principle: Analysis = Lens, Not Filter

  • Raw Source tells you WHAT the code does
  • Enriched Analysis tells you WHY it matters and HOW it fits
  • Both are available to cards — analysis for orientation, source for verification

🔄 Enrichment Flow

1. UPLOAD .dtsx, .py, SQL
→
2. EXTRACT Raw content_md
→
3. ENRICH AI generates analysis
→
4. FEEDBACK Human resolves questions
→
5. APPROVE Analysis ready for cards

⚡ Human Feedback System

When AI encounters something requiring human decision, it creates a Feedback Item with options:

Example: Feedback Item #1

Topic: 7-Hour Deduplication Tie-Breaker Rule

Priority: 🟠 High

Category: Business Rule

Context: Found ROW_NUMBER with 7-hour partition, but tie-breaker logic prioritizes Company A (MRS) without documented justification.

Why It Matters: Incorrect tie-breaker could cause duplicate billing or missed records.

Options:

Option 1: Keep MRS Priority (Document as Business Rule)
✅ No code change | ❌ May perpetuate undocumented assumption

Option 2: Use Timestamp as Tie-Breaker
✅ More transparent logic | ❌ Changes behavior, needs testing

Option 3: Escalate to Business SME ⭐ Recommended
✅ Correct decision with approval | ❌ Delays timeline

📋 Feedback Categories

  • business_rule — Logic decisions needing SME input
  • technical_decision — Architecture choices
  • data_quality — Data issues found
  • security — Credential/access concerns
  • compliance — Regulatory requirements
  • architecture — Design pattern decisions
  • ownership — Who maintains what

🚦 Feedback Priorities

  • 🔴 Critical — Blocks ALL migration work
  • 🟠 High — Blocks specific phase
  • 🟡 Medium — Should resolve, can proceed with assumptions
  • 🟢 Low — Nice to have

⚡ SSIS: Why HIGH VALUE

  • XML complexity — Business logic hidden in encoded SQL
  • Implicit rules — Tie-breakers, dedup logic undocumented
  • Connection mapping — What connects to what and why
  • 2K chars = 10% — Raw excerpt misses 90% of logic

Feedback Needed: Business rule clarifications, undocumented tie-breakers

🔧 Airflow: Why MEDIUM-HIGH

  • Code is readable — But schedule implications aren't obvious
  • DAG structure — Task dependencies need visualization
  • XCom/Variables — Hidden data dependencies
  • Retry logic — Failure handling strategy

Feedback Needed: SLA decisions, alert configuration

📋 Detection Heuristics

# Pseudo-code for enrichment triggering

def should_enrich(artifact: ProjectArtifact) -> tuple[str | None, str]:
    ext = Path(artifact.filename).suffix.lower()
    
    # Always enrich
    if ext == ".dtsx":
        return ("technical_assessment", "SSIS package detected")
    
    # Conditional: Airflow DAG detection
    if ext == ".py" and "from airflow import DAG" in artifact.content_md:
        return ("dag_analysis", "Airflow DAG detected")
    
    # Conditional: Large SQL
    if ext == ".sql" and len(artifact.content_md) > 50_000:
        return ("code_analysis", "Large SQL file (>50K chars)")
    
    # Conditional: Large PDF
    if ext == ".pdf" and artifact.size_bytes > 500_000:
        return ("requirements_extraction", "Large PDF (>500KB)")
    
    # Skip markdown (already readable)
    if ext in (".md", ".txt"):
        return (None, "Already human-readable")
    
    return (None, "No enrichment configured")

Key Insight: Enrichment is for DENSE artifacts

  • SSIS packages are XML soup → need semantic extraction
  • Airflow DAGs are code → but schedule/dependency implications are implicit
  • PDF specs have requirements → but buried in paragraphs
  • Markdown is already consumable → skip enrichment

✅ With Shared Context

  • Connection resolved once, inherited everywhere
  • Business rules catalogued project-wide
  • Clusters batch-migrate together
  • ~500 unique feedback items

🧠 Knowledge Preservation Strategy

For 2,000+ package migrations, we need to accumulate and reuse knowledge. This section explores meta-learning vs hardcoded skills.

The Question: Fine-Tuning?

Approach Pros Cons Verdict
Fine-tuning Deep pattern learning Expensive, black box, frozen knowledge ❌ Not recommended
RAG (Vector DB) Easy updates, explainable Retrieval quality varies ✅ For similar packages
Structured KB Precise, queryable, auditable Manual schema design ✅ For decisions
Technology Profiles Makes generation tech-aware Needs curation ✅ For skill generation

📋 Technology Profiles (Not Hardcoded Skills)

Instead of hardcoding skills like disabled-task-auditor, define technology profiles that the LLM uses during generation:

# technology_profiles/ssis.yaml
structural_patterns:
  - name: "disabled_tasks"
    description: "SSIS packages often contain disabled tasks"
    detection: "DTS:Executable with DTS:Disabled='True'"
    migration_implication: "Must audit before migration"
    skill_suggestion: "Include task audit in discovery"
    
  - name: "change_annotations"
    description: "Developers leave notes in AnnotationLayout"
    detection: "AnnotationLayout elements"
    migration_implication: "Risk information hidden here"
    skill_suggestion: "Parse annotations for risk register"

execution_patterns:
  - name: "incremental_vs_full"
    description: "Many packages support both modes"
    detection: "Control flags like 'processamento_full'"
    migration_implication: "Target must support MERGE and OVERWRITE"

validation_requirements:
  - name: "parallel_run"
    description: "Side-by-side validation before cutover"
    skill_suggestion: "Include parallel run phase"

Result: Skills are generated dynamically from technology characteristics, not hardcoded per-migration.

);

⚠️ Gap Analysis: Multi-Package ETL Migrations

The current system supports SSIS package analysis via artifact uploads and the corp-ssis-analyzer skill, but lacks a structured inventory model for managing multiple packages and their connections.

Current Capabilities

  • ✅ Upload .dtsx files as artifacts (async extraction)
  • ✅ SSIS/T-SQL in tech catalog as "legacy" technology
  • ✅ corp-ssis-analyzer skill for package analysis
  • ✅ corp-databricks-planner skill for migration planning
  • ❌ No structured package inventory
  • ❌ No connection registry across packages
  • ❌ No package-to-package dependency mapping

  • AI proposes: business purpose, criticality

  • Review extracted connections
  • Fill: server, database, auth method
  • Map to target platform connections
  • Flag DSN-only or expression connections

Step 3: Dependency Declaration

  • Select packages this depends on
  • Select packages triggered after
  • System validates: no cycles

  • AI suggests order based on dependencies
  • User adjusts based on business priority
  • Generates migration wave plan

-- New tables for ETL package inventory

CREATE TABLE etl_packages (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    project_id UUID FK → projects ON DELETE CASCADE,
    package_name TEXT NOT NULL,
    package_path TEXT,
    artifact_id UUID FK → project_artifacts,  -- Link to uploaded .dtsx
    business_purpose TEXT NOT NULL,
    execution_schedule TEXT NOT NULL,  -- daily|hourly|weekly|on_demand|event_triggered
    execution_window TEXT,
    avg_duration_minutes INTEGER,
    criticality TEXT NOT NULL DEFAULT 'medium',  -- critical|high|medium|low
    owner_team TEXT,
    migration_priority INTEGER,
    migration_status TEXT NOT NULL DEFAULT 'pending',  -- pending|analyzing|ready|migrating|migrated|verified
    notes TEXT,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    UNIQUE (project_id, package_name)
);

CREATE TABLE etl_connections (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    project_id UUID FK → projects ON DELETE CASCADE,
    connection_name TEXT NOT NULL,
    connection_type TEXT NOT NULL,  -- oledb|flatfile|ftp|odbc|excel|http|smtp|azure_blob|other
    direction TEXT NOT NULL,  -- source|target|both|control
    server TEXT,
    database_name TEXT,
    auth_method TEXT,  -- windows_auth|sql_auth|service_account|key_file|managed_identity
    credential_ref TEXT,  -- Reference to secrets manager, NOT the actual secret
    network_path TEXT,
    migration_target TEXT,  -- Where this maps in target platform
    is_blocker BOOLEAN NOT NULL DEFAULT false,  -- DSN-only, expression-only, etc.
    blocker_reason TEXT,
    notes TEXT,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    UNIQUE (project_id, connection_name)
);

CREATE TABLE etl_package_connections (
    package_id UUID FK → etl_packages ON DELETE CASCADE,
    connection_id UUID FK → etl_connections ON DELETE CASCADE,
    tables_accessed TEXT[],  -- Tables read/written through this connection
    PRIMARY KEY (package_id, connection_id)
);

CREATE TABLE etl_package_deps (
    package_id UUID FK → etl_packages ON DELETE CASCADE,
    depends_on_package_id UUID FK → etl_packages ON DELETE RESTRICT,
    dep_type TEXT NOT NULL DEFAULT 'sequential',  -- sequential|parallel|conditional
    PRIMARY KEY (package_id, depends_on_package_id)
);

CREATE TABLE etl_data_flows (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    package_id UUID FK → etl_packages ON DELETE CASCADE,
    flow_name TEXT NOT NULL,
    source_connection_id UUID FK → etl_connections,
    source_object TEXT,
    target_connection_id UUID FK → etl_connections,
    target_object TEXT,
    transformation_summary TEXT,
    estimated_row_count BIGINT,
    incremental_key TEXT,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
