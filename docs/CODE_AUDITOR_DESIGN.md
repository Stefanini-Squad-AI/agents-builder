# Code Auditor Module — Two Audit Journeys

> Two quality gates: (1) web applications generated by agents/skills, (2) data migration artifacts from the workbench.
> Date: 2026-05-29 | Status: Design
> Cross-cuts: poc-architecture-standards.md, code-quality-system.md, contract-driven-architecture.md, context-integrity-guidelines.md

---

## 1. Problem Statement

Agents generate two kinds of output that need quality gates:

| Output | Example | Standard Source | Current Gap |
|--------|---------|-----------------|-------------|
| **Web applications** | REST APIs, full-stack PoCs, SPAs | poc-architecture-standards.md Cat 1-2 | No automated check |
| **Data migration artifacts** | SSIS parse results, schema dumps, reconciliation runs | poc-architecture-standards.md Cat 6+8 | Reconciliation stubbed, no completeness check |

We have code-quality-system.md (universal review pipeline + fitness functions) and contract-driven-architecture.md (contract taxonomy) but **no module wires them to category-specific standards** for either journey.

**This module** provides two audit journeys, each with its own standards, checks, and scoring — sharing a common engine and data model.

---

## 2. Two Journeys

### Journey 1: Web Application Audit

**When**: Agent generates an API, full-stack app, or SPA
**Standards**: poc-architecture-standards.md Cat 1 (API) + Cat 2 (Full-Stack)
**Checks**: Structural layout, code quality, API contracts, security, architecture fitness, AI failure modes

| What | Why |
|------|-----|
| API backends (FastAPI, Express, Spring Boot) | Cat 1 — primary agent output |
| Full-stack apps (backend + frontend) | Cat 2 — agent-generated PoCs |
| Structural, quality, contract, security, fitness, AI-specific checks | Cat 1-2 mandatory requirements |

**Out of scope**: Data pipelines, infrastructure, event-driven, MCP servers, runtime probing, LLM review (too expensive for gate)

### Journey 2: Data Migration Audit

**When**: Migration workbench produces parse results, schema dumps, reconciliation runs
**Standards**: poc-architecture-standards.md Cat 6 (Pipeline) + Cat 8 (AI-assisted migration)
**Checks**: Parse completeness, schema alignment, reconciliation coverage, context integrity, decision traceability

| What | Why |
|------|-----|
| SSIS parse completeness | 20 known gaps — are they all addressed? |
| Schema dump structural validity | Does the uploaded JSON match DatabaseSchema format? |
| Reconciliation coverage | Are all 5 check types actually implemented (not stubbed)? |
| Context integrity | Do the 8 rules from context-integrity-guidelines.md pass? |
| Decision traceability | Can every LLM proposal be traced to context sources? |

**Out of scope**: Web app checks, infrastructure, runtime DQ checks, Lakebridge connectivity

### 3.2 High-Level Flow

`
workshop audit ./my-app --journey web --category api
workshop audit ./migration-project --journey data
  │
  ▼
AuditService.run(path, journey, category)
  │
  ├─ 1. Load standards YAML for the journey + category
  ├─ 2. Run check groups in priority order:
  │    │
  │    │  Journey 1 (Web App):
  │    ├─ Phase 1: Structural   (filesystem — cheapest)
  │    ├─ Phase 2: Quality      (lint, typecheck, test existence)
  │    ├─ Phase 3: Contract     (OpenAPI, .env.example, Pact)
  │    ├─ Phase 4: Security     (secrets, health endpoint, auth)
  │    ├─ Phase 5: Fitness      (layered deps, no circular deps)
  │    └─ Phase 6: AI-specific  (dead scaffolding, hallucinated APIs)
  │    │
  │    │  Journey 2 (Data Migration):
  │    ├─ Phase 1: Parse completeness   (SSIS gap coverage)
  │    ├─ Phase 2: Schema validity       (DatabaseSchema JSON validation)
  │    ├─ Phase 3: Reconciliation        (stub detection, coverage)
  │    ├─ Phase 4: Context integrity     (8 rules check)
  │    └─ Phase 5: Decision traceability (proposal → context trace)
  │
  ├─ 3. Score findings per standard requirement
  ├─ 4. Generate AuditReport (pass/fail/waivable per requirement)
  ├─ 5. Persist AuditRun + findings to DB
  └─ 6. Return report to CLI (rich table output)
`

### 3.3 Relationship to code-quality-system.md

This module is a **pre-wired orchestration** — not a replacement for the Code Review Agent or Fitness Evaluator MCP servers:

| code-quality-system.md concept | Journey 1 (Web) | Journey 2 (Data) |
|-------------------------------|-----------------|-------------------|
| Phase 1 (static) | structural + quality + security | parse completeness + schema validity |
| Phase 2 (contract) | contract checks | reconciliation coverage |
| Phase 3 (LLM) | **Not used** — too expensive | **Not used** |
| Phase 4 (differential) | **N/A** | context integrity + decision trace |
| Fitness Evaluator | fitness checks (API scope) | fitness checks (migration scope) |
| Service Validation | **Out of scope** | **Out of scope** |

### 4.2 ORM Models

```python
class AuditRun(Base, UuidPkMixin, TimestampsMixin):
    __tablename__ = "code_auditor_runs"
    project_id: Mapped[uuid.UUID] = mapped_column(UUID, ForeignKey("projects.id"))
    app_path: Mapped[str] = mapped_column(Text)
    journey: Mapped[str] = mapped_column(Text)       # AuditJourney
    category: Mapped[str | None] = mapped_column(Text, nullable=True)
    status: Mapped[str] = mapped_column(Text)         # AuditStatus
    total_checks: Mapped[int] = mapped_column(Integer, default=0)
    passed: Mapped[int] = mapped_column(Integer, default=0)
    failed: Mapped[int] = mapped_column(Integer, default=0)
    waived: Mapped[int] = mapped_column(Integer, default=0)
    skipped: Mapped[int] = mapped_column(Integer, default=0)
    score: Mapped[float | None] = mapped_column(Float, nullable=True)
    completed_at: Mapped[datetime | None] = mapped_column(DateTime(timezone=True), nullable=True)
    findings: Mapped[list["AuditFinding"]] = relationship(back_populates="run")
    __table_args__ = (
        CheckConstraint(f"journey IN ({values_csv(AuditJourney)})", name="ck_valid_journey"),
        CheckConstraint(f"status IN ({values_csv(AuditStatus)})", name="ck_valid_status"),
        CheckConstraint("score IS NULL OR (score >= 0 AND score <= 1)", name="ck_score_range"),
    )

class AuditFinding(Base, UuidPkMixin, TimestampsMixin):
    __tablename__ = "code_auditor_findings"
    run_id: Mapped[uuid.UUID] = mapped_column(UUID, ForeignKey("code_auditor_runs.id"))
    check_group: Mapped[str] = mapped_column(Text)
    check_id: Mapped[str] = mapped_column(Text)
    check_name: Mapped[str] = mapped_column(Text)
    severity: Mapped[str] = mapped_column(Text)
    result: Mapped[str] = mapped_column(Text)
    message: Mapped[str] = mapped_column(Text)
    location: Mapped[str | None] = mapped_column(Text, nullable=True)
    suggested_fix: Mapped[str | None] = mapped_column(Text, nullable=True)
    waiver_rationale: Mapped[str | None] = mapped_column(Text, nullable=True)
    waived_by: Mapped[str | None] = mapped_column(Text, nullable=True)
    run: Mapped["AuditRun"] = relationship(back_populates="findings")
    __table_args__ = (
        CheckConstraint(f"check_group IN ({values_csv(CheckGroup)})", name="ck_valid_group"),
        CheckConstraint(f"severity IN ({values_csv(CheckSeverity)})", name="ck_valid_severity"),
        CheckConstraint(f"result IN ({values_csv(CheckResult)})", name="ck_valid_result"),
    )
```

## 7. Check Implementations

### 7.1 Journey 1: Web App Checks

**Structural** (`checks/structural.py`) — pure filesystem, cheapest:

| Detection | Implementation |
|-----------|---------------|
| `directory_exists(paths)` | `all((app_path / p).is_dir() for p in paths)` |
| `file_exists(path)` | `(app_path / path).is_file()` |
| `file_exists_any(paths)` | `any((app_path / p).is_file() for p in paths)` |
| `no_hardcoded_localhost_in_frontend()` | Grep frontend/ for `localhost:\d+` outside config/test |

**Quality** (`checks/quality.py`) — subprocess to existing tools:

| Detection | Implementation |
|-----------|---------------|
| `test_coverage_per_endpoint(min_happy, min_error)` | Parse test files, count per-endpoint vs route definitions |
| `no_hardcoded(patterns)` | Grep source for patterns, exclude .env.example, tests/fixtures |
| `lint_passes()` | Run `ruff check` (Python) or `eslint` (JS/TS) |
| `typecheck_passes()` | Run `mypy` (Python) or `tsc --noEmit` (TS) |

**Contract** (`checks/contracts.py`):

| Detection | Implementation |
|-----------|---------------|
| `endpoint_exists(method, path)` | Parse OpenAPI spec or route definitions |
| `error_response_schema_consistent()` | Parse error responses, verify `{error, code}` shape |
| `frontend_types_from_contract()` | Check src/types/ for generated files |

**Security** (`checks/security.py`):

| Detection | Implementation |
|-----------|---------------|
| `no_secrets_in_source()` | Grep for secret patterns; exclude .env.example |
| `config_from_env_only()` | Find config files; verify env var references |

**Fitness** (`checks/fitness.py`):

| Detection | Implementation |
|-----------|---------------|
| `layered_deps(layers)` | Build import graph, verify direction |
| `no_circular_deps()` | Topological sort on import graph |
| `openapi_coverage()` | Compare route definitions vs OpenAPI spec paths |

**AI-Specific** (`checks/ai_specific.py`):

| Detection | Implementation |
|-----------|---------------|
| `no_dead_scaffolding()` | AST: find unreachable branches, unused routes |
| `no_hallucinated_apis()` | AST: find method calls, verify against known APIs |
| `no_verbose_hollow(threshold, ratio)` | AST: find functions > threshold lines with low logic density |

### 7.2 Journey 2: Data Migration Checks

**Parse Completeness** (`checks/parse_completeness.py`):

| Detection | Implementation |
|-----------|---------------|
| `ssis_parser_gap_coverage(expected)` | Compare known 20 gaps vs current parser capabilities |
| `field_populated(field_path)` | Query DB for parsed packages; check field is not None/empty |

**Schema Validity** (`checks/schema_validity.py`):

| Detection | Implementation |
|-----------|---------------|
| `validate_json_schema(schema_name)` | Pydantic model validation against DatabaseSchema |
| `all_tables_have_columns()` | Iterate schema.tables; fail if any has empty columns |

**Reconciliation** (`checks/reconciliation.py`):

| Detection | Implementation |
|-----------|---------------|
| `not_stubbed(check_type)` | Inspect `_run_check()` — detect hardcoded return values |
| `check_type_count(expected)` | Count implemented check types vs expected |

**Context Integrity** (`checks/context_integrity.py`):

| Detection | Implementation |
|-----------|---------------|
| `decisions_persisted_before_next_item()` | Verify write-back is synchronous before next item |
| `snapshot_has_required_fields()` | Verify profile + dependencies + constraints present |
| `snapshot_rebuilt_per_interaction()` | Verify no cached snapshot reuse |
| `snapshot_within_budget()` | Estimate token count; check within budget |
| `all_patterns_verified_or_flagged()` | Check confidence field on all decisions |
| `output_has_context_citations()` | Parse LLM output for context_citations section |
| `context_bounded_by_cluster()` | Verify snapshot scoped to cluster + dep graph |
| `fallback_exists_for_each_store()` | Verify fallback documented for each knowledge store |

**Decision Trace** (`checks/decision_trace.py`):

| Detection | Implementation |
|-----------|---------------|
| `proposal_has_context_citations()` | Check LLM output includes context_citations |

## 8. Scoring

### 8.1 Per-Check Scoring

| Result | Score |
|--------|-------|
| PASS | 1.0 |
| SKIP | excluded from calculation |
| WAIVED | excluded from calculation |
| FAIL (mandatory) | 0.0 |
| FAIL (non-mandatory) | 0.5 |

### 8.2 Overall Score

```
score = sum(check_scores) / count(applicable_checks)
```

Where applicable = PASS + FAIL (SKIP and WAIVED excluded).

### 8.3 Blocking Thresholds

| Severity | Blocks? | Exit Code |
|----------|---------|-----------|
| CRITICAL fail | Yes — blocks delivery | 1 |
| HIGH fail | Yes — blocks merge (waivable) | 1 |
| MEDIUM fail | No — warn | 0 |
| LOW fail | No — info | 0 |

## 10. Key Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Two journeys | Separate standards + checks, shared engine | Web apps and data migrations have fundamentally different quality criteria |
| Standards as YAML | Machine-readable, version-controlled | Same pattern as technology profiles in context-management-universal.md |
| No LLM in audit loop | Static analysis only | LLM review is expensive and slow; gate must be fast and deterministic |
| Mandatory vs non-mandatory | Mandatory fails block; non-mandatory warns | From poc-architecture-standards.md obrigatorio/recomendado distinction |
| Waiver with rationale | Human must justify | Auditability — every waived finding has a reason and a person |
| Score 0-1 | Simple, comparable across runs | Same scoring as fitness functions in code-quality-system.md |
| Check inheritance | Cat 2 inherits Cat 1 | Full-stack apps include an API backend — don't duplicate checks |
| Journey 2 inspects DB state | Not just filesystem | Migration checks need to query parsed packages, schema dumps, reconciliation runs |
| Per-journey check groups | Different CheckGroup enums | Parse completeness, schema validity, etc. only make sense for data migration |
