1 O problema

Migrações de ETL legados (ex.: SSIS → Databricks) levam meses, envolvem dezenas de pacotes interdependentes, documentação escassa e exigência de paridade absoluta de resultados. O risco é alto, o prazo é apertado, e a visibilidade do progresso para o cliente é frequentemente baixa.
Especificar manualmente o trabalho para agentes de código (Cursor, Claude Code, Gemini CLI) não escala: prompts soltos perdem contexto entre sessões, sem skills reutilizáveis e sem DAG de dependências, agentes repetem trabalho, divergem do padrão e produzem entregas inconsistentes.

2 O que é a plataforma

Um workshop onde equipes técnicas desenham, validam e empacotam o trabalho que será executado por agentes de IA — uma "fábrica de pacotes de trabalho" que padroniza saídas e gera evidência rastreável de cada decisão.
Aplicação web (Next.js 14) sobre API FastAPI, workers Dramatiq, PostgreSQL e Redis. Núcleo agnóstico de LLM (Anthropic / OpenAI / Ollama / Bedrock). Módulos plugáveis registrados via APIRouter; contratos tipados em Pydantic; migrações Alembic.

⭐ ETL Migration Workbench

Migrações ETL exigem mais do que gerar código: exigem descoberta (entender o que existe), contexto compartilhado (regras de negócio, decisões), mapeamento de dependências, geração assistida, reconciliação de paridade e sign-off rastreável. Este módulo materializa cada uma dessas etapas como um estágio dedicado.
Módulo migration_workbench sob /api/migrations, composto por 11 sub-routers independentes que compartilham um único banco de contexto. Cada sub-módulo expõe seu próprio CRUD + serviços e pode ser evoluído sem tocar nos vizinhos.

4.2 O que cada estágio entrega

01 Profiles — Perfis de Tecnologia
Catálogo de combinações origem→destino suportadas (ex.: SSIS → Databricks SQL/PySpark). Garante que o cliente vê desde o início quais cenários a plataforma cobre.
Tabela global (não escopo de projeto) de perfis com origem, destino, famílias de pacotes (data flow, control flow…) e hints de estratégia. Endpoints: GET /profiles.
02 Registry — Registro de Pacotes
Inventário oficial dos pacotes a migrar. Cada pacote tem dono, prioridade, estado de processamento. Importação em massa (bulk) para evitar trabalho manual em projetos com centenas de pacotes.
CRUD completo + bulk import + estatísticas (POST /:project/packages/bulk, GET /:project/packages/stats). Cada pacote tem ID estável, metadados e referência para artefatos fonte.
03 Context — Contexto Compartilhado
Conexões (origem/destino), regras de negócio extraídas durante a descoberta e decisões arquiteturais. Tudo o que todos os cards devem respeitar mora aqui, com versionamento e autoria.
Três entidades: connections, business_rules, decisions. Endpoints separados para cada uma, com lookup rápido para enriquecer outros estágios.
04 Analysis — Análise dos Pacotes
Para cada pacote: detecção automática de padrões de dados (lookup, surrogate keys, SCD), estratégia recomendada de migração e bloqueadores conhecidos antes de gerar qualquer linha de código.
Pipeline interno: extractor → analyzer → data_pattern_classifier → strategy_classifier → blockers. Parsers plugáveis por tipo de artefato (.dtsx, .json, .sql).
05 Map — Mapa de Migração
Visão única que mostra a ordem em que os pacotes devem ser migrados, quem depende de quem e onde estão os portões humanos de aprovação.
DAG construído com NetworkX, persistido e exposto via API. Suporta arestas explícitas (declaradas) e implícitas (derivadas da análise).
06 Propagation — Propagação de Contexto
Quando uma regra ou decisão muda, todos os cards afetados são marcados automaticamente. Evita esquecimentos e mantém o projeto coerente conforme o entendimento evolui.
Job que percorre o DAG a partir do nó alterado e atualiza vetores de contexto downstream. Idempotente; histórico mantido.
07 Skills — Skills do Workbench
Conjunto de regras específicas de migração (paridade de hash, controles de auditoria, naming de notebooks Databricks…) que o agente deve seguir em todo pacote.
Estende a biblioteca de skills do núcleo com skills tipadas para migração. Versionadas por projeto, exportadas no .agents/.
08 Generation — Geração de Cards
A partir do pacote analisado, da estratégia escolhida e do contexto acumulado, a IA produz o design e os cards Jira prontos para execução, incluindo entregáveis esperados.
Endpoints /packages/:id/preview, /generate, /download, /design. Geração assíncrona via Dramatiq; saída persistida em data/projects/<id>/artifacts/.
09 Documentation — Documentação Viva
Cada decisão, mapeamento e card vira página navegável. Cliente e auditoria veem o mesmo material, sempre atualizado, sem depender de confluência paralela.
Render Markdown server-side com cross-refs resolvidos; cache por revisão de conteúdo.
10 Reconciliation — Reconciliação
Compara resultados antes e depois da migração (linha-a-linha, hash, agregados). Sem reconciliação, não há paridade comprovada — e sem paridade, não há migração aceita.
Harness de comparação parametrizável, sidecars de teste (Python + JSON de resultado). Endpoint para registrar runs e resultados.
11 Sign-off — Aprovação
Trilha auditável de aprovação por critério de aceitação, com autor, timestamp e justificativa. É o aceite formal que destrava a próxima fase.
API de sign-off persistindo decisões com referência ao commit fonte (futuramente: PR review do GitHub espelhado, propriedades no Jira).

7 Arquitetura técnica

Aplicação em camadas com fronteiras claras; cada módulo do Workbench é independente e plugável; provedor de LLM é abstrato; armazenamento dividido entre PostgreSQL (estado), Redis (filas) e sistema de arquivos (artefatos).
  • LLM — provedores plugáveis (4 implementados, base abstrata).
  • Módulos — padrão de módulo-dentro-do-workshop: cada vertical (como Workbench) é um pacote isolado com seu próprio router/schemas/services.
  • Parsers de artefato — registrados por tipo de arquivo na etapa de Analysis.
  • Webhooks — receptores Jira e GitHub previstos no plano de integrações.
